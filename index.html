Why this matters

Understanding the JSONB null propagation problem.

The root cause

In PostgreSQL, jsonb_agg() returns NULL when it aggregates zero rows — not an empty array. When this null is passed directly into jsonb_set(), it silently overwrites the target field with null, corrupting data.

In your schema structure (schema → sections[] → fields[]), any section that has an empty or missing fields key will cause the inner jsonb_agg to return NULL, which then propagates upward and nullifies the entire section's fields.

The dangerous pattern

jsonb_set(
  section,
  '{fields}',
  -- ⚠ If section has no fields, this returns NULL
  (SELECT jsonb_agg(...)
   FROM jsonb_array_elements(section->'fields') AS field)
)

The safe pattern

jsonb_set(
  section,
  '{fields}',
  -- ✓ Falls back to original value if agg returns NULL
  COALESCE(
    (SELECT jsonb_agg(...)
     FROM jsonb_array_elements(section->'fields') AS field),
    section->'fields'
  )
)

Defence-in-depth layers

1
COALESCE every jsonb_agg — fall back to original value
2
schema IS NOT NULL — skip rows with null columns
3
jsonb_typeof = 'array' — ensure sections/fields are arrays before iterating
4
EXISTS filter — only touch rows that actually have the target key
5
BEGIN / COMMIT — wrap in a transaction so you can ROLLBACK
